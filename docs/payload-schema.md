# `case-review-payload` v0.7

Single payload emitted by the **Atom Console** plugin (`case_review_atom_mvp`,
build `atom-mvp v0.17.17-ux-fixes`) and consumed by the
`nsa-atom-precedent-review` skill (v1.5), `hhb-atom-precedent-review` (v1.1),
or `hra-atom-precedent-review` (v1.1).

> **Scope (current):** `policy_area` \u2208 `{"NSA", "HHB", "HRA"}`. The plugin loads ATLAS Atoms +
> ATLAS Atom Rules from the AI Policy Rule Book Bitable; the legacy V0.3 Rule
> Book is intentionally not carried.

> **History:** v0.1 (initial design, in `REVIEW_AND_DESIGN.md`) was superseded by
> v0.6 in Phase 5. v0.6 added `_route_hint`, `claims_slice`, `rule_book_ref`,
> and `engine_simulation`, and dropped `engine_result` / `candidate_claim` in
> favor of letting the skill re-derive against the sliced claim set.
> v0.7 removed `claims_slice` from the wire payload (claims baked into skill
> Appendix B), added `structural_stability` with counterfactual + fragility,
> and added multi-vertical routing (`_route_hint` now varies per issue scope).

## Top-level shape

```json
{
  "schema": "case-review-payload",
  "version": "0.7",
  "_route_hint": "nsa-atom-precedent-review",
  "policy_area": "NSA",
  "candidate":             { /* see § Candidate */ },
  "rule_book_ref":         { "schema": "atom-catalog", "version": "v1.0", "policy_area": "NSA" },
  "engine_simulation":     { /* see § Engine simulation */ },
  "structural_stability":  { /* see § Structural Stability */ },
  "precedents":            [ /* see § Precedent */ ]
}
```

| field                 | type                   | notes                                                                                          |
|-----------------------|------------------------|------------------------------------------------------------------------------------------------|
| schema                | const                  | `"case-review-payload"`                                                                        |
| version               | const                  | `"0.7"`                                                                                        |
| _route_hint           | string                 | `"nsa-atom-precedent-review"` / `"hhb-..."` / `"hra-..."` — route signal for skill router      |
| policy_area           | string                 | `"NSA"` / `"HHB"` / `"HRA"`                                                                   |
| candidate             | Candidate              | the case under review                                                                          |
| rule_book_ref         | RuleBookRef            | catalog version pointer, used for the skill's sanity check                                     |
| engine_simulation     | EngineSimulation       | deterministic engine output for the reviewer's draft atoms (hint, not the verdict)             |
| structural_stability  | StructuralStability?   | engine stability + counterfactual fragility (null if engine unavailable)                        |
| precedents            | Precedent[]            | top-K, ranked by hybrid BM25 + atom-Jaccard; capped at 8                                       |

## Candidate

```json
{
  "item_id": "...",
  "record_id": "...",
  "policy_title": "AN",
  "issue": "NSA",
  "visual_summary": "...",
  "asr_text": "...",
  "ocr_text": "...",
  "caption_text": "...",
  "case_url": "https://...",
  "images_attached_count": 4,
  "images_note": "jpg×4",
  "composite_image_url": "https://p-mira-img-sign-...png",
  "draft_atom_tags": {
    "C": ["atom.foo"], "S": [...], "P": [...], "R": [...],
    "E": [...], "M": [...], "X": [...]
  }
}
```

**`case_url`** is optional and populated when the Pending Cases table
includes a `case_url` text field (URL pointing to the moderation tool page
with full clips and case context). When present, the plugin renders a
clickable link in the case-meta panel so the reviewer can open the full case
in a new tab.

Atom keys use canonical dotted-namespace form (`atom.not_applicable`,
`subject.adult`, `modifier.medical_context`). `serial_code` strings
(`"C37 — Not applicable"`) appear in the AtomTagger UI but are never sent in
the payload.

`composite_image_url` is **optional** and only populated by the prefill
scheduler path: a Mira-hosted PNG (composite of up to 12 keyframes, 2-col
grid, ~7d TTL) generated by the backend so the skill can fetch one image
rather than relying on a chat-pasted clipboard image. Manual chat-paste
flows leave this field absent. The skill's image-handling rule: prefer
chat-attached image; if absent and `composite_image_url` is present, fetch
it; otherwise fall back to `visual_summary` text-only.

## ClaimsSlice (removed in v0.7)

> As of payload v0.7, `claims_slice` is **no longer included** in the wire payload.
> Non-approve claims are baked into each specialist skill's Appendix B (~109–252
> claims depending on vertical). The local engine still uses the full claims corpus
> internally, but does not serialize it to the payload. This reduced typical payload
> size from ~211 KB to ~15–30 KB.

## RuleBookRef

```json
{
  "schema": "atom-catalog",
  "version": "v1.0",
  "policy_area": "NSA"
}
```

Used by the skill to sanity-check that the in-prompt atoms_catalog Appendix
matches the version assumed by the plugin. A mismatch surfaces in `watch_outs`
but does not abort.

## EngineSimulation

Deterministic engine result for the reviewer's *draft* atom tags. The skill
treats this as a hint and re-derives after re-tagging.

```json
{
  "verdict": "VIOLATION",
  "applied_claim_chain": ["AN.001"],
  "fired_claims": [
    {
      "claim_id": "AN.001",
      "outcome_base": "VIOLATION",
      "outcome_after_lifts": "VIOLATION",
      "lift": null,
      "specificity": 4
    }
  ],
  "suppressed": [],
  "routing": {
    "non_realistic_routes_to": null,
    "cross_title_redirect": null
  }
}
```

`verdict ∈ { "VIOLATION", "NOT_RECOMMEND", "NOT_FOR_FEED", "APPROVE",
"NO_CLAIM_FIRED" }`.

## Structural Stability

Computed by the plugin's two deterministic analysis modules: `computeStability`
(DI-like consistency check) and `computeCounterfactual` (Phase B fragility
analysis). Present when `engine_simulation` produces a non-null result; `null`
otherwise.

```json
{
  "DI": 1.0,
  "verdict_consistent": true,
  "highest_claim_outcome": "VIOLATION",
  "counterfactual": {
    "structurally_fragile": true,
    "min_flips_to_inverse": 1,
    "nearest_flip_path": [
      {
        "action": "remove",
        "dim": "C",
        "atom_key": "atom.some_key",
        "resulting_verdict": "APPROVE"
      }
    ]
  },
  "fragility_score": 0.0
}
```

| field                                  | type     | notes                                                                              |
|----------------------------------------|----------|------------------------------------------------------------------------------------|
| DI                                     | number   | Defensibility Index: 1.0 = verdict aligns with expected outcome; 0.0 = misaligned |
| verdict_consistent                     | boolean  | `true` when engine verdict matches/exceeds severity threshold for the decision     |
| highest_claim_outcome                  | string   | highest-severity outcome from any fired claim (before lifts)                       |
| counterfactual                         | object?  | Phase B structural fragility analysis (null if unavailable)                        |
| counterfactual.structurally_fragile    | boolean  | `true` when a single gate-atom flip (C/S/P/R/E) inverts the engine verdict        |
| counterfactual.min_flips_to_inverse    | number   | `1` if fragile, `Infinity` if robust                                              |
| counterfactual.nearest_flip_path       | array?   | `[{action, dim, atom_key, resulting_verdict}]` or `null` if robust                |
| fragility_score                        | number   | `0.0` = fragile (1-flip inverts), `1.0` = robust (no single gate flip inverts)    |

### Fragility semantics

**Only gate-level atoms (C/S/P/R/E) are tested.** M/X atoms are excluded because
modifier_excluded and exception atoms exist by design as rule-book escape paths.
Testing them would flag nearly every non-approve case, measuring rule-book
completeness rather than genuine content ambiguity.

Strategies:
1. **Remove each tagged atom** — can removing one gate atom suppress the winning claim?
2. **Near-miss fill** — claims missing exactly 1 gate dimension: can adding that atom trigger a new verdict?

True content-grounded ambiguity assessment is done skill-side (see verdict schema
`ambiguity_assessment` field). The skill emits this object in its verdict:

| field                         | type   | notes                                                                 |
|-------------------------------|--------|-----------------------------------------------------------------------|
| classification                | string | `"L1"` / `"L2"` / `"L3"` — overall ambiguity tier                    |
| reasoning                     | string | free-text explanation of ambiguity drivers                            |
| precedent_support             | string | `"strong"` / `"partial"` / `"none"` — how well precedents back the call |
| ~~confidence~~                | ~~int~~| **DEPRECATED (v0.18.0)** — removed from skill output; signals absorbed into classification + precedent_support |

> `confidence` was a 1–5 integer that conflated decision certainty with content
> ambiguity. It offered no actionable insight beyond what `classification` +
> `precedent_support` already convey and has been removed.

## Precedent

```json
{
  "rank": 1,
  "case_id": "case_xxx",
  "policy_title": "AN",
  "decision": "flag",
  "score": 0.82,
  "score_text": 0.71,
  "score_atom": 0.94,
  "atoms": {
    "C": [...], "S": [...], "P": [...], "R": [...],
    "E": [...], "M": [...], "X": [...]
  },
  "visual_summary": "...",         // capped 1500 chars
  "asr_text": "...",                // capped 800
  "ocr_text": "...",                // capped 800
  "caption_text": "...",            // capped 800
  "rationale": "...",               // capped 1000
  "images": [{ "name": "...", "type": "jpg", "size": 12345 }]
}
```

Capped fields use the marker `…[truncated +N chars]` when truncated. Caps are
enforced by the plugin's `_cap()` helper (constants `PREC_VS_CAP=1500`,
`PREC_TEXT_CAP=800`, `PREC_RAT_CAP=1000`). Per-precedent payload size is
typically 4–6 KB; the full payload is 30–45 KB end-to-end.
